# 循环工程一期：项目目录、节点注册与调度 Implementation Plan

> **执行说明：** 每次只执行一个 Task。若环境具备 `superpowers:subagent-driven-development` 或 `superpowers:executing-plans`，可以使用；不具备这些技能时，直接严格遵循本 Task 的 checkbox、Red→Green、验证和提交步骤。缺少技能不得成为跳步、伪造验证或扩大范围的理由。

**Goal:** 让项目配置可版本化发布，让 Linux、macOS、Windows 节点通过一次性邀请码取得设备身份并持续心跳，再把 `QUEUED` 缺陷原子调度到唯一 GitLab Runner。

**Architecture:** `project_profile` 的已发布版本在任务创建时完整快照；节点以 X.509 设备证书访问节点 API；调度器先做硬过滤和稳定排序，再用 PostgreSQL reservation 占用槽位，最后触发带唯一 Runner tag 的中央 Pipeline。

**Tech Stack:** Spring Boot 4.1、PostgreSQL 17、Flyway、Bouncy Castle、GitLab Runners/Pipelines API v4、JUnit 5、Testcontainers、WireMock。

  active_slots integer not null default 0,
  allowed_projects_json jsonb not null,
  capabilities_json jsonb not null,
  last_heartbeat_at timestamptz null,
  created_at timestamptz not null default now(),

);
create table node_heartbeat (
  node_id uuid not null references repair_node(id),

  metrics_json jsonb not null,

);

create table repair_task (
  id uuid primary key,
  defect_id uuid not null references defect(id),
  defect_revision bigint not null,
  project_key varchar(128) not null,
  profile_revision bigint not null,
  profile_snapshot_json jsonb not null,
  base_sha char(40) not null,
  state varchar(32) not null,
  priority integer not null default 100,
  requested_node_id uuid null references repair_node(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

create table task_reservation (

create index ix_task_schedulable on repair_task(priority, created_at) where state = 'QUEUED';

- [ ] **Step 3: Implement device-bound node APIs**

Require a verified client certificate for `/api/node/v1/**`. Extract node ID from `CN=repair-node:{uuid}`, compare certificate serial and public-key hash with `repair_node`, and reject revoked, expired or disabled devices.

Configure the node-facing TLS listener with `server.ssl.client-auth: want`, a server key store and a trust store containing only the device CA. If a corporate load balancer terminates mTLS, it must remove any inbound certificate headers and the application must accept those headers only from the load balancer's mutually authenticated address.

Heartbeat request every 15 seconds:

```json
{
  "observedAt": "2026-07-18T08:00:00Z",
  "appliedRevision": 2,
  "runner": {"status": "online", "version": "18.2.0"},
  "slots": {"active": 1, "limit": 4},
  "resources": {"cpuPercent": 26.1, "memoryAvailableBytes": 8589934592, "diskAvailableBytes": 53687091200},
  "tools": {"os": "linux", "arch": "amd64", "java": "21.0.8", "maven": "3.9.11", "node": "24.18.0", "pnpm": "10.14.0", "opencode": "1.0.180"},
  "activeAttemptIds": ["9a606751-5a0c-43ba-8cd6-07bb64729e74"],
  "lastError": null
}
```

State calculation: disabled → `DISABLED`; drain requested → `DRAINING`; last heartbeat older than 45 seconds → `OFFLINE`; runner down or disk below 10 GiB → `DEGRADED`; active slots > 0 → `BUSY`; otherwise `ONLINE`. A scheduled sweep applies this calculation every 15 seconds.

Sort ascending by this tuple, with node UUID as final tie-breaker:

```java
public record NodeScore(
    int requestedPenalty,
    int projectOwnerPenalty,
    int cachePenalty,
    int failureRatePermille,
    int loadPermille,
    Instant lastAssignedAt,
    UUID nodeId) implements Comparable<NodeScore> {}
```

Use penalty 0 for a preference match and 1 otherwise. Failure rate uses the latest 50 finished attempts; nodes with no history use 500. Load is `activeSlots * 1000 / concurrencyLimit`. `lastAssignedAt` sorts oldest first.

- [ ] **Step 3: Implement reservation before external I/O**

Within one transaction, select one queued task using:

```sql
select id from repair_task
where state = 'QUEUED'
order by priority asc, created_at asc
for update skip locked limit 1;
```

Lock the selected node, recheck `active_slots < concurrency_limit`, increment slots, insert an `ACTIVE` reservation expiring at `now()+120 seconds`, move the task to `RESERVED`, and commit. Then trigger the central project Pipeline with variables `LOOP_TASK_ID`, `LOOP_RESERVATION_ID`, `LOOP_NODE_ID`, `LOOP_RUNNER_TAG`. Persist the returned Pipeline ID. A definite GitLab 4xx compensates by expiring the reservation and returning the task to `QUEUED`; timeout, connection reset or 5xx is ambiguous and changes the reservation to `TRIGGER_UNKNOWN` without releasing capacity.

`PipelineTriggerReconciler` lists recent central-project Pipelines and reads their variables to find the unique `LOOP_RESERVATION_ID`. If found, persist its Pipeline ID and restore `ACTIVE`; if no matching Pipeline exists after two minutes and two successful GitLab queries, release the reservation. Never send a second trigger request for the same reservation.

`ci/repair-pipeline.yml`:

```yaml
stages: [repair]
repair:
  stage: repair
  tags: ["$LOOP_RUNNER_TAG"]
  timeout: 2h
  interruptible: true
  script:
    - repair-executor run --server "$LOOP_CONTROL_PLANE_URL" --task "$LOOP_TASK_ID" --reservation "$LOOP_RESERVATION_ID"
  artifacts:
    when: always
    expire_in: 14 days
    paths:
      - out/change.patch
      - out/result-manifest.json
      - out/events.jsonl
      - out/test-results/
```

Run `ReservationReaper` every 15 seconds. For an `ACTIVE` reservation older than its 120-second expiry with no attempt, query the recorded Pipeline. Cancel a created/pending Pipeline before returning the task to `QUEUED`; if GitLab reports a running Job, extend the reservation for 60 seconds and let attempt bootstrap reconcile it; if GitLab is unreachable, mark `EXPIRY_PENDING` and do not reassign. Only a confirmed cancelled/failed/missing Pipeline may release the node slot.

## Plan 03 completion gate

- Project revisions are immutable and a task keeps the exact selected JSON snapshot.
- Invite codes expire after 10 minutes, are single-use and never stored in plaintext.
- Node identity is bound to its device key; model credentials are never requested or transmitted.
- Heartbeat interval is 15 seconds; 45 seconds silence produces `OFFLINE`.
- Concurrency is remotely configurable from 1 to 10 and enforced by both reservation and Runner configuration.
- Scheduler emits eligibility/rejection reasons and has deterministic ranking.
- Two scheduler instances cannot overbook a node, and a failed Pipeline call releases its reservation.
