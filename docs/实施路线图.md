# 循环工程一期实施路线图

> **执行说明：** 每次只执行一个 Task。若环境具备 `superpowers:subagent-driven-development` 或 `superpowers:executing-plans`，可以使用；不具备这些技能时，直接严格遵循本 Task 的 checkbox、Red→Green、验证和提交步骤。缺少技能不得成为跳步、伪造验证或扩大范围的理由。

**Goal:** 按六个可独立验收的阶段交付“GitLab Issue → 信息补充 → 节点修复 → 单测 → 中央发布 MR → 待测试”的一期闭环。

**Architecture:** 单体仓库包含 Spring Boot 控制面、Vue 3 管理台、Go 节点运行时和版本化 JSON 契约。GitLab Issues、Webhook、Runner 与 Job Artifacts 提供外部协作和执行底座；PostgreSQL 保存权威状态、Outbox 和租约。

**Tech Stack:** Java 21、Spring Boot 4.1.0、Maven、PostgreSQL 17、Flyway、Vue 3、TypeScript、Vite、Node.js 24 LTS、Go 1.26、GitLab Runner、OpenCode。

---

## 1. 为什么拆成六份计划

设计规格包含六个可以单独测试和演示的子系统。每份计划完成后都必须形成一个可运行增量，并提交独立 Git commit；不得跨计划提前实现后续能力。

| 顺序 | 计划 | 独立验收结果 | 依赖 |
|---|---|---|---|
| 01 | [仓库骨架与控制面基础](./实施计划-01-仓库骨架与控制面基础.md) | Java、Vue、Go、PostgreSQL 和 CI 均可运行 | 无 |
| 02 | [GitLab 缺陷入口与状态机](./实施计划-02-GitLab缺陷入口与状态机.md) | Issue Webhook 可幂等进入待补充或排队 | 01 |
| 03 | [项目目录、节点注册与调度](./实施计划-03-项目目录节点注册与调度.md) | 节点可注册、心跳、匹配并触发定向 Pipeline | 01、02 |
| 04 | [repair-node、Runner 与 OpenCode 执行器](./实施计划-04-节点运行时与OpenCode执行器.md) | 节点可执行样例修复并上传结构化结果 | 01、03 |
| 05 | [Artifact、Publisher、MR 与状态同步](./实施计划-05-Artifact发布MR与状态同步.md) | 候选补丁可安全发布 MR 并转待测试 | 02、04 |

边界规则：

- `contracts/v1` 是 Java 与 Go 之间唯一共享的数据契约；跨语言不得复制未版本化 DTO。
- 控制面按业务能力分包，不创建横跨所有能力的 `service`、`util`、`manager` 大包。
- `repair-node` 只负责节点治理；`repair-executor` 只负责单次任务执行。
- Publisher 在控制面内运行，但使用独立 package 和独立 GitLab 机器人凭据。
- 前端按 feature 切分，禁止把所有请求放进一个 `api.ts`。

## 3. 版本与开发基线

- Java：21 LTS；Maven Toolchains 明确选择 JDK 21。
- Spring Boot：4.1.0。
- Maven：3.9.x。
- PostgreSQL：17.x；本地和测试使用 Testcontainers。
- Node.js：24 LTS；包管理器固定 pnpm，并提交 `pnpm-lock.yaml`。
- Go：`go 1.26`；CI 安装当期 1.26 安全补丁版本。
- GitLab：REST API v4；不得依赖跨项目写入 `CI_JOB_TOKEN`。
- OpenCode：节点上报实际版本；项目 Profile 可配置允许的最低版本；一期测试固定一份已验证版本清单。

## 4. 跨计划工程规则

### 4.1 TDD

每个行为按以下顺序实现：

1. 写一个只验证单一行为的失败测试。
2. 运行精确测试并确认失败原因正确。
3. 编写使该测试通过的最小实现。
4. 运行精确测试，再运行所属模块完整测试。
5. 提交一个只包含该行为的 commit。

### 4.2 数据库

- 所有 schema 变更使用新的 Flyway migration；已提交 migration 不允许修改。
- 业务状态更新和 Outbox 写入必须位于同一数据库事务。
- 领取任务使用 `FOR UPDATE SKIP LOCKED` 与带过期时间的 reservation。
- 时间统一保存为 UTC `timestamptz`，API 使用 ISO-8601。

### 4.3 安全

- 日志、事件和错误体不得出现 GitLab Token、设备私钥或模型密钥。
- 节点 API 同时校验设备身份与任务短期令牌。
- OpenCode 子进程使用显式环境白名单，不继承完整 Runner 环境。
- Publisher 在全新临时目录应用补丁，验证 Base SHA 和禁止路径后才推送。

### 4.4 提交格式

使用 Conventional Commits：

```text
build: initialize control-plane module
feat(defect): persist GitLab issue deliveries
feat(node): enroll repair nodes
feat(executor): stream OpenCode events
feat(publisher): create idempotent merge requests
feat(web): show repair task timeline
test(e2e): cover duplicate webhook recovery
docs: add node operations runbook
```

## 5. 阶段门槛

- 计划 01 完成前，不接真实 GitLab。
- 计划 02 完成前，不注册真实节点。
- 计划 03 完成前，不执行 OpenCode。
- 计划 04 完成前，不给 Publisher 写权限。
- 计划 05 完成前，不设置 `repair::ready-for-test`。
- 计划 06 完成前，不宣称 Windows 正式支持。

## 6. 全量验证命令

从仓库根目录执行：

```powershell
mvn -pl control-plane test
pnpm --dir console-web test --run
pnpm --dir console-web build
go -C node-runtime test ./...
git diff --check
```

预期结果：所有命令退出码为 0；Java、Vue 和 Go 测试均无失败；前端生成生产构建；Git diff 无空白错误。
