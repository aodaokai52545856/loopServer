# Loop Engine 威胁模型（一期）

范围覆盖控制面、Publisher、个人 repair-node、OpenCode 与内网 GitLab。目标：在一期边界内明确信任假设与缓解，而不是宣称原生本机执行可完全沙箱化。

## 信任假设

| 资产 | 信任级别 | 说明 |
|---|---|---|
| 内网 GitLab | 可信基础设施 | Issue/Pipeline/Artifact/MR 权威源；Webhook 校验 secret 并限制来源 |
| 项目代码边界 | 有条件可信 | 仅授权内部可信仓库到个人节点；不可信项目需隔离节点（增强项，非一期硬要求） |
| 控制面 PostgreSQL | 权威状态 | 业务更新与 Outbox 同事务 |
| Publisher 机器人身份 | 高敏感 | 唯一可推分支/创建 MR 的写身份；Token 加密、最小范围、定期轮换 |

## 威胁与缓解

### 恶意 Issue 文本

- 风险：提示注入、路径诱导、伪造验收叙述。
- 缓解：结构化字段解析；OpenCode 仅改工作区；待测试硬门槛（非空安全 Patch、必选测试、Artifact/manifest、分支/commit/MR、完整事件）缺一不可。

### 恶意 Artifact

- 风险：篡改 Patch、测试证据或 manifest。
- 缓解：Artifact 摘要校验；失败则禁止 Publisher；保留结构化结论与链接。

### 被攻陷的节点

- 风险：泄漏模型密钥、伪造事件、滥用 Runner。
- 缓解：设备证书 + 短期单任务令牌；节点无仓库写 Token；OpenCode 启动前清除 GitLab/任务凭据；管理台可 Drain/禁用/轮换证书。

### 失窃邀请码

- 风险：未授权节点加入。
- 缓解：一次性邀请、过期检查、仅存储 SHA-256；证书签发失败不消耗邀请语义按实现保证；管理台仅展示一次明文。

### 失窃 Publisher 凭据

- 风险：向目标仓库写入恶意分支/MR。
- 缓解：服务端保管、最小项目范围、轮换演练；节点侧永不持有该 Token。

### 依赖脚本与原生宿主限制

- 风险：恶意 Maven 插件、npm lifecycle、本地原生代码逃逸。
- 缓解：只授权可信内部仓库；日志脱敏；不可信项目使用容器隔离节点作为增强选项。原生本机无法完全隔离，必须在运维策略中显式接受该残余风险。

## 平台标签

- Linux / macOS：正式支持。
- Windows：Beta，直到服务模式与本机 AI 工具兼容性认证完成。
