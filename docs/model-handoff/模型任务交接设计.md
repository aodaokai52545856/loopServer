# 循环工程一期：外部模型任务交接设计

**状态：** 已实施并验证

**日期：** 2026-07-18

**适用对象：** Grok 4.5 及其他可读取完整工作区、但不具备 Superpowers 技能的代码模型

## 1. 目标

将现有六份实施计划中的 44 个 Task 转换为可逐个交给外部模型执行的受控工作单元，降低以下风险：

- 模型一次读取全部文档后混淆当前范围。
- 跳过 Red→Green 测试过程，直接编写大范围实现。
- 自行替换已经确定的组件、状态名、数据库结构或安全边界。
- 修改当前 Task 之外的文件，或者提前实现后续 Task。
- 遇到文档与代码冲突时自行猜测。
- 只根据文字说明声称完成，没有测试和 Git 证据。

成功标准是：不依赖特定模型技能，每次仅提供一个 Task，模型能够准确完成预检、实现、验证和单 Task commit；不确定时安全停止；完成后能够由另一个模型或人工独立复核。

## 2. 前提和范围

### 2.1 前提

- 执行模型能够读取完整 Git 工作区及其中的 Markdown 文件。
- 设计文档、实施计划和交接文件会一起复制到执行模型的工作区。
- 每次对话只执行一个 Task。
- 执行模型没有 `superpowers:*` 技能，也不要求安装这些技能。
- Git 仓库是任务状态和变更证据的权威来源。

### 2.2 本设计包含

- 模型中立的执行总则。
- 不可变架构决策摘要。
- 44 个 Task 的索引和独立任务卡。
- 动态启动提示词模板与 PowerShell 生成工具。
- 两阶段启动确认。
- 完成报告和独立复核模板。
- 对现有计划中 Superpowers 专用说明的兼容性修改。

### 2.3 本设计不包含

- 自动调用 Grok 或其他模型的 API。
- 自动接受模型提交、合并分支或推送远端。
- 多个模型并行修改同一个工作区。
- 替代现有架构设计和实施计划中的技术内容。
- 根据模型偏好重新选择框架、依赖或状态机。

## 3. 目录结构

交付目录固定为：

```text
model-handoff/
├── 00-模型执行总则.md
├── 01-不可变架构决策.md
├── 02-任务索引.md
├── 03-启动提示词模板.md
├── 04-完成报告模板.md
├── 05-任务复核提示词.md
├── 06-人工操作手册.md
├── tools/
│   ├── New-ModelTaskPrompt.ps1
│   └── Test-ModelHandoff.ps1
├── task-cards/
│   ├── LE-P01-T01.md
│   ├── LE-P01-T02.md
│   └── LE-P06-T07.md（共 44 张）
└── generated/
    └── .gitkeep
```

`generated/` 中的动态提示词不进入 Git；任务卡和模板必须进入 Git。复制到实际开发仓库时，`model-handoff/`、架构设计、路线图和六份实施计划应保持相对路径不变。

## 4. 标识和任务粒度

Task ID 格式为 `LE-P{计划号}-T{任务号}`：

- `LE-P01-T01` 表示计划 01 的 Task 1。
- `LE-P03-T04` 表示计划 03 的 Task 4。
- ID 一旦发布不得复用或重新编号。

每张卡只对应实施计划中的一个 `### Task N`。Task 内的五个 Step 是一个不可拆分提交单元。模型不得在同一对话中开始下一张卡，即使当前任务提前完成。

## 5. 上下文分层和读取顺序

### 5.1 每次必须读取

1. `model-handoff/00-模型执行总则.md`。
2. `model-handoff/01-不可变架构决策.md`。
3. 当前 Task 的任务卡。
4. 任务卡引用的实施计划 Task 章节。
5. 任务卡列出的架构设计章节。
6. 当前 Task 涉及的现有代码和测试。

### 5.2 按需读取

- 当前 Task 的直接前置任务卡。
- 当前代码显式引用的契约或数据库 migration。
- 任务卡列出的操作、安全或协议文档。

### 5.3 默认不得加载

- 与当前 Task 无关的其他五份实施计划全文。
- 后续 Task 的实现步骤。
- 无关模块的源代码。

任务卡提供导航和约束，不复制实施计划中的大段代码。实施计划仍是具体实现步骤的唯一权威来源，任务卡负责消除范围、前置条件和停止条件的歧义。

## 6. 规则优先级与冲突处理

规则解释顺序如下：

1. 当前任务启动提示词中经人工填写的明确变更。
2. `01-不可变架构决策.md`。
3. 当前任务卡。
4. 当前实施计划的 Task 章节。
5. 正式架构设计。
6. 实施路线图中的通用规则。
7. 当前代码中的既有模式。

任务卡原则上只能澄清计划，不能静默覆盖计划。任意两层发生实质冲突时，模型不得自行按优先级“猜一个结果”，必须返回 `BLOCKED`，列出冲突原文和需要人工决定的问题。只有启动提示词中的明确人工决定可以解除该冲突。

## 7. 任务卡固定结构

每张 `task-cards/LE-Pxx-Txx.md` 必须包含：

1. **身份信息：** Task ID、名称、计划、阶段。
2. **唯一目标：** 一句话、可验证的目标。
3. **明确不做：** 当前任务不得涉及的后续能力。
4. **前置条件：** 前置 Task ID 及应存在的文件、表、接口或测试。
5. **必须阅读：** 精确文件路径和章节标题。
6. **开始状态：** 分支、Base commit、工作区和基线测试要求。
7. **允许修改：** 文件或目录白名单。
8. **禁止修改：** 不得触碰的文件、migration、接口和模块。
9. **不可变约束：** 与本 Task 直接相关的状态名、协议、安全和版本决定。
10. **执行顺序：** Preflight、Red、Green、精确测试、模块测试、Git 检查、commit。
11. **验收标准：** 命令、退出码和行为断言。
12. **停止条件：** 当前 Task 特有条件和全局停止条件引用。
13. **完成报告：** 必须返回的证据字段。

“允许修改”应尽可能使用具体文件；只有计划本身要求一组同类文件时才允许目录级白名单。模型发现需要白名单外修改时必须停止，不能先改后解释。

## 8. 无 Superpowers 的兼容规则

现有计划头部引用了 `superpowers:subagent-driven-development` 和 `superpowers:executing-plans`。这些技能不是外部模型的前置条件。

所有计划头部将增加模型中立说明：

- 如果执行环境具有这些技能，可以使用。
- 如果没有，直接人工遵循任务中的 checkbox、TDD、验证和提交步骤。
- 缺少技能不能成为跳过任何步骤的理由。
- 不得伪造技能调用、子代理审查或验证结果。

交接总则必须完整描述需要遵循的行为，不能只引用某个模型专用能力。

## 9. 两阶段启动协议

### 9.1 阶段一：预检

第一次发送动态启动提示词后，模型只能读取和执行只读检查，不得编辑文件、安装依赖或提交。返回格式固定为：

```text
状态：READY 或 BLOCKED
Task ID：
我理解的唯一目标：
明确不做：
前置条件检查：
必须读取的文件：
允许修改：
禁止修改：
计划执行的测试：
停止条件：
发现的冲突：
```

只有人工确认内容正确并发送“开始执行 Task LE-Pxx-Txx”后，模型才能进入阶段二。确认消息中的 Task ID 必须与预检一致。

### 9.2 阶段二：执行

执行顺序固定为：

1. 再次确认 HEAD 和工作区与预检一致。
2. 编写计划指定的失败测试。
3. 运行精确测试，确认按计划中的原因失败。
4. 编写使该测试通过的最小实现。
5. 运行精确测试。
6. 运行任务卡要求的模块级验证。
7. 检查 `git diff --check`、修改白名单和敏感信息。
8. 只暂存当前 Task 的文件。
9. 创建一个当前 Task 的 commit；不得 amend 既有 commit。
10. 输出完成报告；不得继续下一个 Task。

## 10. 动态提示词生成

`tools/New-ModelTaskPrompt.ps1` 的调用方式：

```powershell
./model-handoff/tools/New-ModelTaskPrompt.ps1 -TaskId LE-P03-T04 -Model Grok-4.5
```

动态提示词包含：

- Task ID 和模型名称。
- 仓库绝对路径与当前分支。
- 当前 HEAD/Base commit。
- `git status --short` 结果。
- 最近五个 commit。
- 前置任务及其预期产物。
- 任务卡、实施计划和架构章节路径。
- 是否允许当前模型创建 commit。
- 人工填写的已知偏差。
- 阶段一固定输出格式。

`git status` 路径和最近提交消息按不可信数据展示并明确禁止作为指令解释，避免文件名或 commit message 污染模型提示词。

工具默认要求工作区干净。存在未提交修改时拒绝生成；人工可以通过显式参数声明允许的脏路径和原因，但生成内容必须逐项列出这些路径。工具不得自动清理、暂存、提交或修改业务文件。

生成文件写入 `model-handoff/generated/{TaskId}-{UTC时间}.md`，其内容同时打印到标准输出。工具应兼容 Windows PowerShell 7，并避免依赖网络或第三方模块。

## 11. 全局停止条件

出现任一条件，模型必须停止并输出 `BLOCKED`：

- 启动时存在未声明的工作区修改。
- Base commit、分支或前置 Task 与启动提示词不一致。
- 前置 Task 应产生的文件、表、接口或测试不存在。
- 计划要求的 Red 测试没有失败，或失败原因与预期不同。
- 继续执行必须修改白名单之外的文件。
- 任务卡、实施计划、架构设计和实际代码发生实质冲突。
- 需要改变状态名、API、数据库表、组件边界、安全策略或依赖版本。
- 外部 API 或依赖版本无法根据工作区锁文件、官方文档或现有测试确认。
- 需要删除数据、强制推送、重写历史、覆盖用户修改或输出凭据。
- 精确测试通过，但模块测试出现无法归因于当前 Task 的失败。
- 同一阻塞问题连续尝试三次仍没有新的诊断证据。
- 当前环境无法运行必选验收命令。

阻塞报告格式：

```text
状态：BLOCKED
Task ID：
已完成步骤：
阻塞位置：
实际证据：
计划预期：
需要人工决定的问题：
工作区修改：
建议选项：
```

## 12. 权限和安全边界

执行模型可以：

- 读取完整工作区。
- 修改当前任务卡白名单内的文件。
- 运行任务卡要求的非破坏性测试、构建和 Git 只读检查。
- 在所有验收通过后创建一个当前 Task 的 commit。

执行模型不得：

- `push`、merge、rebase、reset、force、删除分支或改写历史。
- 清理或覆盖启动前已经存在的用户修改。
- 修改已经提交的 Flyway migration。
- 输出、提交或上传 Token、模型密钥、设备私钥和证书私钥。
- 安装未在计划中批准的依赖或自行升级版本。
- 勾选后续任务、修改正式设计，或声称整个阶段完成。
- 将无法运行的测试报告为通过。

## 13. 完成报告

执行完成后固定返回：

```text
状态：COMPLETED
Task ID：
Base commit：
产生的 commit：
修改文件：
Red 测试命令、退出码和关键失败：
Green 测试命令、退出码和结果：
完整模块验证：
git diff --check：
未执行的验证及原因：
与计划的偏差：
残余风险：
```

只要存在必选验证未执行、计划偏差未批准或工作区仍含未提交的当前 Task 修改，状态就不能是 `COMPLETED`。

## 14. 独立复核

每个执行 commit 完成后，使用 `05-任务复核提示词.md` 交给另一个模型或人工。复核者只读检查，不直接修复，必须验证：

- Commit 父节点等于启动时的 Base commit，或偏差已经声明。
- Commit 只包含当前 Task 白名单文件。
- 实现与任务卡、计划和不可变架构决策一致。
- Red→Green 证据可信，关键测试可重新运行。
- 模块验证和 `git diff --check` 通过。
- 没有凭据、危险命令、无关重构或后续 Task 实现。
- 失败恢复、幂等、安全边界和平台要求没有被削弱。

复核结果只能是：

- `ACCEPTED`：允许开始下一 Task。
- `REWORK`：列出必须由原 Task 修复的具体问题。
- `BLOCKED`：需要架构或人工决策。

复核不得把“看起来正确”当作测试证据，也不得在同一复核步骤顺手修改代码。

## 15. 状态与连续执行

`02-任务索引.md` 记录 Task ID、计划章节、前置 Task、任务卡链接和人工状态。状态取值：

- `PENDING`
- `READY`
- `IN_PROGRESS`
- `BLOCKED`
- `REVIEW`
- `ACCEPTED`

执行模型不得修改索引状态；状态由人工或外部编排器在检查报告和 commit 后更新。只有前置 Task 全部为 `ACCEPTED`，下一 Task 才能进入 `READY`。

## 16. 实施和验收范围

本设计落地时必须：

1. 生成 44 张任务卡，数量和计划 Task 完全对应。
2. 每张任务卡能定位到唯一计划文件和唯一 Task 标题。
3. 每张卡包含白名单、前置条件、验收命令和停止条件。
4. 修改七份实施文档的代理说明，使其不依赖 Superpowers。
5. 提供可手工复制的启动模板，即使脚本不可用也能工作。
6. 提供 PowerShell 生成工具及其成功、未知 Task、脏工作区测试。
7. 提供独立复核模板和人工操作手册。
8. 自动检查任务卡数量、ID 唯一性、链接存在性、任务标题一致性和 Markdown 围栏。

只有以上检查全部通过，才能把模型交接包标记为可用。
